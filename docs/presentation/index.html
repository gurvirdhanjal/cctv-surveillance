🔑 Key Design Decision

  • • Separate OS processes — Python GIL isolation
  • • Frames via Shared Memory, not Redis
  • • Redis only carries 24-byte metadata pointer
  • • Zero-copy frame transport to GPU
  • • Each ingestion worker: independent failure domain

📡 Message Bus

  • • frames:groupA/B/C/D — 4 ingestion streams
  • • detections — GPU output stream
  • • tracking + alerts — Identity output
  • • dead_letter — poison message sink
  • • XAUTOCLAIM recovers unACKed after 10s

🛡️ Resilience

  • • Camera disconnect: exponential backoff 1→60s
  • • Worker crash: supervisor restart <5s
  • • Redis outage: 200-frame in-memory buffer/cam
  • • DB failure: 50k row buffer → JSON overflow
  • • Stale frames (>200ms old) silently dropped
Why not microservices? Single GPU server → separate containers gain nothing. Modular pipeline gives process isolation + restart semantics without the overhead. Kafka upgrade path is documented for future multi-server scale.

Per-Frame Processing Flow

1 Read SHM slot → validate seq_id > last_seen AND age <200ms
2 SCRFD detects all faces → discard <40px or blur <25
3 YOLOv8n detects persons → ByteTrack assigns stable local_track_id
4 AdaFace embeds valid face crops → 512-dim L2-normalised vector
5 Publish tracklet to detections stream (no raw pixels)

GPU Efficiency

  • Batched inference — frames from multiple cameras in one SCRFD/YOLOv8 call
  • ONNX + CUDA — all 4 models use CUDAExecutionProvider
  • TensorRT optional — YOLOv8n can use TRT engine for 3× speedup
  • CPU fallback — automatic when GPU is unavailable
  • One ByteTracker per camera — independent state, no cross-contamination

04 — Re-Identification Engine

Cross-Camera Identity Matching

The Re-ID Problem

When a person moves from Camera A to Camera B, they get a new local_track_id. The system must recognise it's the same person and assign the same global_track_id.

60
Cameras
→
O(n²)
Naive Problem
→
O(k)
With Pre-filter

Three-Layer Filtering Strategy

Layer 1 — Zone Adjacency
Only compare against tracklets in same or adjacent zones. Cuts search space by ~80%.
Layer 2 — Time Window
Only compare against tracklets seen in the last 5 minutes. Eliminates stale entries.
Layer 3 — FAISS Cosine Search
Flat IP index on 512-dim embeddings. sim ≥ 0.65 AND margin ≥ 0.08 → same person.

Match Decision Logic

Condition Decision Action
sim ≥ 0.72 AND margin ≥ 0.08 Known Person Assign person_id
sim ≥ 0.65 AND margin ≥ 0.08 Re-ID Match Reuse global_track_id
sim < 0.65 OR margin < 0.08 New Track New UUID assigned
person_id IS NULL > 500ms UNKNOWN Alert fired

Guard Correction + Audit Trail

  • Guard spots wrong identity → corrects via UI
  • Correction event → corrections Redis stream
  • Identity Service cascades update to ALL tracking_events with that global_track_id
  • WebSocket broadcasts track_corrected to all connected guards
  • Every match logged to reid_matches table with confidence, cameras, source

Homography → Floor Position

  • Pre-computed 3×3 H matrix per camera (stored as JSON in MSSQL)
  • Bounding-box centre projected → floor_x, floor_y
  • 4-point calibration wizard with <2px reprojection threshold
  • Null stored if camera not yet calibrated

06 — Alert System

Finite State Machine Alert Engine

Three Alert Types — FSM Behaviour

Alert TypeEntrySustainCooldownReset Condition
UNKNOWN_PERSON No person_id > 500ms 60s/zone Identity confirmed
PERSON_LOST Absent all cams > 30s 120s Re-detected anywhere
CROWD_DENSITY Count > max_cap > 10s 300s/zone Count drops below

Event-Time Evaluation

  • FSM evaluates on timestamp_ms — frame capture time, NOT arrival order
  • Out-of-order events handled correctly up to 5s late window
  • Events >5s late are discarded to prevent phantom alerts
  • Dedup key: (alert_type, zone_id) within 60s window
  • Max 10 active alerts per zone — frontend groups the rest

Alert Lifecycle — Four States

ACTIVE Entry condition met and sustained → alert fired and pushed to guard
↓
ACKNOWLEDGED Guard sees and accepts the alert via PATCH /api/alerts/{id}/acknowledge
↓
RESOLVED Reset condition met or guard manually resolves

WebSocket Event Push

  • alert_fired — pushed immediately, no throttle
  • person_location — throttled to 5fps, diff-only
  • track_corrected — immediate on identity fix
  • camera_snapshot — every 2s per subscribed camera
  • All events fan out via Redis pub/sub ws_events across API replicas

Client Reconnect Safety

On WebSocket reconnect, client calls GET /api/state/snapshot before processing any diff events — guarantees consistent state even after long disconnects.

Frame to Alert — 8-Step Journey

1
Camera Captures Frame
Ingestion Worker decodes via OpenCV → raw BGR ndarray
2
Write to Shared Memory
16-byte header (seq_id + timestamp_ms) + raw frame bytes → SHM_Slot
3
Publish Frame Pointer to Redis
{cam_id, shm_name, seq_id, timestamp_ms} — 24 bytes only, no pixels in Redis
4
GPU Inference
SCRFD → YOLOv8 → ByteTrack → AdaFace embed. Stale >200ms → drop + XACK
5
Publish Tracklet to Detections Stream
{cam_id, local_track_id, bbox, embedding, timestamp_ms} per confirmed track
6
Identity Resolution
Zone adjacency filter → FAISS cosine search → global_track_id + person_id assigned
7
Homography + Alert FSM
Project to floor_x/y → FSM evaluates entry/sustain/cooldown conditions
8
DB Write + WebSocket Push
DB Writer batches to MSSQL. FastAPI pushes via Redis pub/sub to all browser clients.

Frame Transport Design Rationale

❌ Naive Approach
~2 MB
per 1080p frame in Redis
✅ Our Approach
24 B
SHM pointer in Redis

Shared memory eliminates network serialisation entirely. The GPU process reads directly from the memory segment — zero copy.

Event Envelope — All Streams

{ "schema_version": "1", "seq_id": 1247983, "timestamp_ms": 1714048234521, "cam_id": "cam_07", "shm_name": "vms_cam07_frm", ...payload... }

Backpressure Strategy

  • • Each cam slot holds 1 frame at a time — latest wins
  • • Stale frames (>200ms) dropped before GPU touch
  • • Redis streams capped at MAXLEN 500
  • • Lag >500 → admin alert fires automatically

09 — Reliability

Production-Grade Error Handling

Failure Recovery Matrix

Failure Recovery
Camera RTSP disconnect Backoff 1→2→4→8→60s; 3 fails → mark is_active=false
Ingestion worker crash Supervisor restart <5s; XAUTOCLAIM recovers after 10s
GPU out-of-memory Catch OOM → halve batch size → retry; on crash: restart
Redis unavailable 200-frame buffer/cam; retry every 2s; drain on recovery
MSSQL write failure 3 retries → 50k-row ring buffer → JSON file overflow
Re-ID false match Guard corrects via UI → cascade DB update + WS broadcast
Poison message (3× unACKed) Moved to dead_letter stream; original XACK'd
WebSocket disconnect Socket.io auto-reconnect + GET /api/state/snapshot rehydrate
Bad homography calibration Wizard rejects save if reprojection error ≥ 2px

⚠️ Degraded Mode — Redis Outage

📷 Ingestion: buffer 200 frames/cam in-memory, retry every 2s
🎮 GPU: runs locally, ephemeral UUIDs, Re-ID disabled
🗄️ DB Writer: JSON file (100MB cap, oldest dropped)
💻 Frontend: displays "System degraded — Re-ID offline" banner

📊 Observability Stack

  • Prometheus metrics per process: latency p50/p95/p99, frames_dropped, stream lag, DB queue, GPU util
  • Worker heartbeat: Redis key heartbeat:{worker_id} with 15s TTL, refreshed every 10s
  • Heartbeat expiry → admin alert within 15 seconds
  • Grafana dashboard: GPU, Redis lag, MSSQL queue depth
  • GET /api/health exposes all worker status, Redis, MSSQL

🔐 Security

  • • JWT auth, 8-hour expiry — all endpoints except /health
  • • Role guards: guard / manager / admin
  • • Zone + camera access per user (row-level)
  • • Expired token → HTTP 401; insufficient role → HTTP 403
